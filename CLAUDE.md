Context and conventions for AI-assisted development on this repo.

## What this repo is

`win_dell` is a fully unattended Windows 10/11 post-install automation framework.
A single `irm | iex` one-liner triggers it; everything after that ‚Äî Windows Update
cycles, power settings, debloat, registry tweaks, Dell app installs, Tailscale
registration, and cleanup ‚Äî runs automatically across reboots until complete.

Target hardware: Dell laptops/desktops, Windows 10 21H2+ and Windows 11 22H2+.
All code must run on **Windows PowerShell 5.1**. PowerShell 7 is not assumed.

## One-liner

```powershell
irm "https://raw.githubusercontent.com/karolperkowski/win_dell/main/install.ps1" | iex
```

Re-running on an existing install automatically updates scripts and preserves state.
No parameters, no flags, no prompts. `irm | iex` works because install.ps1 needs
no arguments ‚Äî the update-vs-fresh decision is made internally based on whether
the repo already exists on disk.

## PowerShell 5.1 rules ‚Äî enforce these on every change

These are the most common mistakes when writing code that targets PS 5.1:

| Forbidden | PS 5.1 alternative |
|---|---|
| `$x ?? $y` | `if ($x) { $x } else { $y }` |
| `$x?.Property` | `if ($x) { $x.Property }` |
| `??=` | `if (-not $x) { $x = $y }` |
| `[array].Count` on pipeline result | `@($result).Count` |
| `ConvertFrom-Json -AsHashtable` | Use the `ConvertTo-Hashtable` shim in `State.psm1` |
| `Get-WindowsUpdate -AcceptAll` | `-AcceptAll` is only valid on `Install-WindowsUpdate` |
| `Import-Module foo` | Always add `-DisableNameChecking -Force` for internal modules |

Run this before every commit to catch PS 5.1 issues:
```powershell
grep -rn '??' core/ *.ps1 --include="*.ps1" --include="*.psm1"
```

## Shared constants ‚Äî always use $WD.*

`core/Config.psm1` exports a single `$WD` object. Every script that needs a path
or task name imports this module and reads from `$WD`:

```powershell
Import-Module (Join-Path $PSScriptRoot 'Config.psm1') -DisableNameChecking -Force

$WD.DeployRoot      # C:\ProgramData\WinDeploy
$WD.StateFile       # C:\ProgramData\WinDeploy\state.json
$WD.LogDir          # C:\ProgramData\WinDeploy\Logs
$WD.TailscaleJson   # C:\ProgramData\WinDeploy\tailscale.json
$WD.TaskResume      # WinDeploy-Resume
$WD.TaskMonitor     # WinDeploy-Monitor
$WD.TaskNotify      # WinDeploy-Notify
$WD.StageOrder      # @('WindowsUpdate', 'PowerSettings', ...)
$WD.StageLabels     # @{ WindowsUpdate = 'Windows Update', ... }
$WD.RebootAllowedStages
```

**Never hardcode `C:\ProgramData\WinDeploy` in any script.**
**Never duplicate the stage list ‚Äî it lives only in `Config.psm1`.**

## Adding a new stage

1. Create `core/YourStage.ps1` using an existing stage as a template.
2. The script must accept `-StageName` and `-Config` params and return:
   `@{ Status = 'Complete'|'RebootRequired'|'Failed'; Message = 'string' }`
3. Add the stage name to `$WD.StageOrder` in `Config.psm1` at the correct position.
4. Add a label to `$WD.StageLabels` in `Config.psm1`.
5. Add the stage to `$Script:STAGE_SCRIPTS` in `Orchestrator.ps1`.
6. Add a config entry in `config/settings.json` under `"Stages"`.
7. If the stage may reboot, add it to `$WD.RebootAllowedStages` in `Config.psm1`.

## Stage return contract

Every stage script must return exactly one of:

```powershell
@{ Status = 'Complete';      Message = 'Human readable.' }
@{ Status = 'RebootRequired'; Message = 'Why reboot is needed.' }
@{ Status = 'Failed';        Message = $_.Exception.Message }
```

The orchestrator handles all three. Stages must not call `Restart-Computer`
directly ‚Äî they return `RebootRequired` and the orchestrator reboots.

## State file

`C:\ProgramData\WinDeploy\state.json` ‚Äî never edit by hand on a live deployment.

Key fields:
- `CurrentStage` ‚Äî next stage to run
- `CompletedStages` ‚Äî array of finished stage names
- `RebootCount` ‚Äî incremented before every reboot (max 10, then orchestrator aborts)
- `DeployComplete` ‚Äî set true by `Set-DeployComplete` in final Cleanup stage

All reads/writes go through `State.psm1` functions. Never `Get-Content` the state
file directly in a stage script ‚Äî except `WindowsUpdate.ps1`'s `Read-StateRaw`
which is a justified bypass for writing sub-keys.

## Scheduled tasks

Five tasks, each with a specific purpose and lifecycle:

| Task | Runs as | Window | Registered by | Removed by |
|---|---|---|---|---|
| `WinDeploy-Resume` | SYSTEM | Hidden | Resilience.psm1 | Cleanup.ps1 |
| `WinDeploy-Monitor` | Users | Visible | Resilience.psm1 | Cleanup.ps1 |
| `WinDeploy-Notify` | Users | Hidden | Resilience.psm1 | Notify.ps1 (self) |
| `WinDeploy-AutoLogonSafety` | SYSTEM | Hidden | Resilience.psm1 | After 6h (self-deletes) |
| `WinDeploy-Watchdog` | SYSTEM | Hidden | Resilience.psm1 | Never (permanent) |

`WinDeploy-Notify` intentionally survives Cleanup so it fires on the first clean boot.
`WinDeploy-AutoLogonSafety` disables auto-logon unconditionally 6 hours after bootstrap,
regardless of whether Cleanup ever runs ‚Äî protects against deployment getting stuck.
`WinDeploy-Watchdog` runs every 30 minutes and kills any Orchestrator process older than 4 hours.

**Task self-healing:** `Resilience.psm1::Assert-ScheduledTasks` is called at the top
of every bootstrap and orchestrator run. It re-registers any missing tasks automatically.
Deleting tasks manually is fully recoverable by re-running `irm ... | iex`.

## Logging

Use `Write-LogInfo`, `Write-LogSuccess`, `Write-LogWarning`, `Write-LogError`
from `Logging.psm1`. Call `Initialize-Logger -Stage 'StageName'` at the top
of every stage script and `Close-Logger -FinalStatus 'SUCCESS'|'FAILED'` before
every return.

Never use `Write-Host` in stage scripts ‚Äî it bypasses the log file.

## Configuration

All deployment behaviour is controlled by `config/settings.json`. Stage scripts
read their config from the `$Config` hashtable passed in by the orchestrator.
Never hardcode behaviour that an operator might want to change.

Key config sections:
- `Stages.<StageName>.ContinueOnError` ‚Äî whether orchestrator continues past a failure
- `Tailscale.AuthKey` ‚Äî set this to skip the QR flow entirely
- `Tailscale.QrTimeoutMinutes` ‚Äî how long to wait for a QR scan (default 30)
- `Debloat.RemoveOptional` ‚Äî set true to also remove the optional app list
- `WinTweaks.RunWinUtil` ‚Äî set false to skip WinUtil and only run direct registry tweaks

## Manifest and integrity verification

`manifest.json` and `manifest.sig` are generated automatically by
`.github/workflows/manifest.yml` on every push to main. The Action:
1. Captures the current commit SHA
2. Downloads the branch ZIP and computes its SHA-256
3. Writes `manifest.json` with `commit_sha`, `zip_url` (using commit SHA, not branch name), `zip_sha256`
4. Signs it with HMAC-SHA256 using the `MANIFEST_SIGNING_KEY` GitHub secret
5. Commits both files back

`install.ps1` verifies the manifest signature before downloading anything.
The signing key is embedded in `install.ps1` as `$MANIFEST_SIGNING_KEY` and
must match the `MANIFEST_SIGNING_KEY` GitHub secret. If you rotate the key,
update both places.

**The GitHub Actions workflow requires the `workflow` scope on the PAT.**
The manifest ZIP URL uses the commit SHA (`/archive/{sha}.zip`) not the branch
name ‚Äî commit SHA URLs are immutable and bypass GitHub's CDN cache.

## Common failure points

| Symptom | Cause | Fix |
|---|---|---|
| `Unexpected token '??'` | PS 7 syntax in a PS 5.1 script | Replace with `if/else` |
| `The property 'Count' cannot be found` | `.Count` on a scalar from pipeline | Wrap in `@()` |
| Unapproved verb warning on import | Function name uses non-standard verb | Rename using `Get-Verb` approved list |
| Old version served by `irm` | GitHub CDN cache | Use `?t=<timestamp>` cache-buster |
| Reboot prompt during automation | Missing `$ConfirmPreference = 'None'` or `-Confirm:$false` | Add both |
| Monitor window not showing | Task running as SYSTEM | Monitor task must run as `BUILTIN\Users` |
| Tailscale QR not appearing | `tailscale.json` not written yet | Check Tailscale stage logs |
| `Get-WindowsUpdate` parameter error | `-AcceptAll`/`-IgnoreReboot` on query cmdlet | These only work on `Install-WindowsUpdate` |
